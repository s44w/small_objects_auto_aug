# Уроки из реальной сборки ВКР — что НЕ делать

> Эти 22 правила извлечены из 26 fix-задач, заведённых в beads при подготовке защищённой ВКР. Каждое правило — конкретная ошибка, которую агент совершил и которую пришлось чинить пост-фактум. **Скилл `vkr-author` обязан предотвращать их в новых проектах.**

---

## A. Ошибки на уровне markdown (исправлять в text/*.md)

### A.1 Тире — только en-dash `–`, никогда em-dash `—`
- **Контекст:** при свободном написании агент склонен ставить em-dash `—` в пояснениях. По СТО СГАУ везде только короткое тире `–` (en-dash, U+2013).
- **Как избежать:** запретить em-dash в стайл-гайде, в audit-проверке `grep '—' text/*.md` должно вернуть 0.
- **Источник:** `mf-8ir`.

### A.2 Жирный `**...**` запрещён в основном тексте
- **Контекст:** агент использовал жирный для выделения "Цель работы.", "Задачи." внутри глав — для академического СТО СГАУ это нарушение.
- **Исключение:** только в `text/00-titlepage.md` (Реферат) для prefix-ов "**Выпускная квалификационная работа бакалавра:**", "**Презентация:**".
- **Источник:** `mf-axo`.

### A.3 Декоративные `[N]` в конце абзаца запрещены
- **Контекст:** агент ставил `[5]` в конце абзаца "для галочки" без конкретного утверждения из источника.
- **Как делать правильно:** *"в работе [5, табл. 3] показано что Chamfer Distance составляет 0,142 на ShapeNet/cars"*. Конкретные цифры/факты.
- **Источник:** `mf-pwf`.

### A.4 У каждой таблицы — ссылка `[N]` или пометка "собственная разработка"
- **Контекст:** часть таблиц подписана как `Таблица 1 – Сравнение методов` без указания откуда данные.
- **Как правильно:** `Таблица 1 – Сравнение методов [4]` (если из источника) или `Таблица 1 – Конфигурация стенда (собственные данные автора)`.
- **Источник:** `mf-6jd`.

### A.5 Цитата `[0]` запрещена — источников 0 не бывает
- **Контекст:** при перенумерации источников образовалась ссылка [0] — явный баг.
- **Как избежать:** audit-проверка `grep '\[0\]' text/*.md` должна возвращать 0.
- **Источник:** `mf-ezm`.

### A.6 ГОСТ 7.32/7.1/7.0.5 не упоминать в основном тексте
- **Контекст:** агент написал во Введении "Работа оформлена в соответствии с требованиями ГОСТ 7.32-2017 [1]..." — это формальные стандарты оформления, в дипломе на них ссылаться неуместно.
- **Исключение:** ГОСТ Р ИСО/МЭК 25010-2015 (модель качества ПО) — он по теме, оставляем.
- **Источник:** `mf-bkz`.

### A.7 Реферат — строго по структуре Маршуниной
Формат:
1. **`+ЗаголРеферСодерж`**: заголовок "Реферат" (ПРОПИСНЫМИ через w:caps).
2. Жирный prefix: `**Выпускная квалификационная работа бакалавра:** N с., M рисунков, K таблиц, L источников, P приложений.`
3. Жирный prefix: `**Презентация:** X слайдов Microsoft PowerPoint.`
4. Стиль `+КлючСловаРеферат`: ключевые слова ПРОПИСНЫМИ через запятую (5-10 терминов).
5. 4 коротких абзаца: тема исследования / объект / цель работы (через "Цель работы – ...") / что рассмотрено и какие результаты получены.
- **Источник:** `mf-fop`, `mf-arf`.

### A.8 В Реферате не оставлять мета-инструкции
- **Контекст:** в `text/00-titlepage.md` остались строчки "Стиль заголовка: +ЗаголРеферСодерж...", которые попадали в финальный docx.
- **Как избежать:** все правила оформления держать в `docs/style-guide.md` или в комментариях `<!--...-->`.
- **Источник:** `mf-arf`, `mf-soa`.

### A.9 Введение ~1 страница, Заключение ~1 страница
- **Контекст:** агент писал 4-5 страниц введения с длинными описаниями актуальности, что превышает академические нормы.
- **Целевые объёмы:** Введение 200-350 слов, Заключение 250-350 слов.
- **Источник:** `mf-e06`.

### A.10 Проза → таблицы/перечни/формулы
- **Контекст:** в главе про требования каждое НФТ описано отдельным длинным абзацем — должна быть сводная таблица "Атрибут | Метрика | Целевое значение | Источник".
- **Когда применять:** decision-таблицы стека технологий, сводные таблицы методов, требования по ISO 25010.
- **Источник:** `mf-z4s`, `mf-dir`.

### A.11 Большие JSON-блоки/листинги кода → диаграммой
- **Контекст:** контракт сообщения Kafka в виде сырого JSON в академическом тексте смотрится плохо.
- **Как правильно:** перевести в class-диаграмму или таблицу полей. Сырой код — только в Приложении.
- **Источник:** `mf-mpc`.

### A.12 alt-text у изображений всегда пустой
- **Контекст:** `![Архитектура системы](path)` — Pandoc генерирует второй caption из alt-text помимо нашей правильной подписи "Рисунок 1 – ...".
- **Как избежать:** только `![](path)`. Audit: `grep '!\[[^]]' text/*.md` = 0 совпадений.
- **Источник:** `mf-p22`, `mf-oi6`.

### A.13 Заголовки спецразделов — Title Case в markdown
- **Контекст:** `# ВВЕДЕНИЕ` сохраняет ПРОПИСНЫЕ как raw text. В TOC отображается ПРОПИСНЫМИ, что ломает оформление Маршуниной.
- **Как правильно:** `# Введение` — ПРОПИСНЫМИ становится в docx через `w:caps` стиля `+ЗАГОЛОВОК по центру`. В TOC — Title Case.
- **Источник:** `mf-7ot`, `mf-9v1`.

### A.14 Без цифровой нумерации в тексте заголовков
- `# 1 АНАЛИТИЧЕСКИЙ ОБЗОР...` — нет. `# Аналитический обзор...` — да. Нумерация добавится автоматом стилем `+Заголовок 1 уровня`.
- **Источник:** общая конвенция Маршуниной.

### A.15 Сквозная нумерация рисунков и таблиц по работе
- Рисунок 1, 2, 3, ... 17 (а не 4.1, 4.2). Таблица 1, 2, ..., 14.
- **Источник:** общая конвенция Маршуниной.

### A.16 У каждой таблицы — упоминание в тексте
- **Контекст:** определено 14 таблиц, упомянуто только 12. Читатель должен быть направлен к таблице фразой "в таблице N показано..." до её появления.
- **Как избежать:** audit проверяет соответствие `grep -c '^Таблица'` и `grep -c 'таблиц[еау] [0-9]'`.
- **Источник:** `mf-a1u`.

### A.17 Длинные абзацы > 200 слов разбивать
- **Контекст:** в архитектурной главе попадались абзацы по 238 слов — академически плохо читается.
- **Целевой максимум:** 150-200 слов на абзац.
- **Источник:** `mf-52q`.

### A.18 Не строить нарратив на "методике + прогнозе по литературе"
- **Контекст:** изначально ML-глава была "методика и прогнозируемые улучшения". Преподавателю не понравилось — переписали на "обучение проведено, метрики получены".
- **Терминология:** заменить на "получено", "измерено", "эмпирически подтверждено". Запретить "прогноз*", "ожидае*", "литературные данные".
- **Источник:** `mf-j41`.

## B. Ошибки на уровне Pandoc-сборки (исправляются в pipeline, агенту знать не нужно)

### B.1 Code blocks разноцветные
- **Симптом:** Pandoc по умолчанию подсвечивает синтаксис в code blocks. В академической работе должно быть моноширинно-чёрно.
- **Фикс:** `--syntax-highlighting=none` в `build-docx.sh`.
- **Источник:** `mf-cx6`.

### B.2 alt-text дублируется как caption
- **Фикс:** `--from=markdown-implicit_figures` отключает auto-caption Pandoc.
- **Источник:** `mf-p22`, `mf-oi6`.

### B.3 Списки рендерятся с bullet-маркерами
- **Симптом:** Pandoc делает Word's "List Paragraph" с точками. По СТО СГАУ должны быть обычные абзацы с дефисом в начале.
- **Фикс:** в `apply-stosgau-styles.lua` функции `BulletList` и `OrderedList` разворачивают список в плоские параграфы стиля `+Абзац с отступом 1-ой строки` с en-dash префиксом.
- **Источник:** `mf-1ys`, `mf-sb4`.

### B.4 Таблицы без границ
- **Фикс:** в постпроцессе устанавливается стиль "Table Grid" (или нативный) на каждую таблицу.
- **Источник:** `mf-svu`.

### B.5 Compact-стиль внутри ячеек
- **Симптом:** Pandoc для ячеек таблиц вшивает служебный стиль `Compact`, не входящий в шаблон Маршуниной.
- **Фикс:** в `postprocess-docx.py` массовая замена `Compact` → styleId `989` (= `+Текст в таблице`).
- **Источник:** `mf-svu`.

### B.6 Перенос таблиц без повтора шапки
- **Симптом:** многостраничная таблица переносится без визуальной "Продолжение таблицы N".
- **Фикс:** в `postprocess-docx.py` устанавливается `<w:tblHeader/>` на первой строке каждой таблицы + `<w:cantSplit/>` на каждой строке. Word нативно повторяет шапку при переносе.
- **Источник:** `mf-kyp`, `mf-4mj`, `mf-7he`.
- **История попыток:** первая попытка — вручную вставлять параграф "Продолжение таблицы N" — провалилась (вставлялась даже там где таблица не переносилась). Второй заход через `tblHeader` — корректное native решение.

### B.7 Word диалог "Обновить поля?" при открытии docx
- **Симптом:** при открытии docx Word предлагает обновить fields (TOC, INCLUDEPICTURE).
- **Фикс:** в `postprocess-docx.py` массовая замена `w:dirty="true"` → `w:dirty="false"` через regex по document.xml.
- **Источник:** `mf-0kz`.

### B.8 TOC показывает "Table of Contents" вместо "СОДЕРЖАНИЕ"
- **Фикс:** Pandoc флаг `--metadata=toc-title:"СОДЕРЖАНИЕ"`. Дополнительно `postprocess-docx.py` меняет pStyle TOC-заголовка на `+ЗаголРеферСодерж`.
- **Источник:** `mf-9v1`, `mf-7ot`.

### B.9 TOC не заполнен (Word ленится при первом открытии)
- **Симптом:** в TOC только заголовок без содержимого, либо при открытии нужно нажать F9.
- **Фикс:** гибрид — после Pandoc-сборки временную копию открыть через Word COM (`update-fields.ps1` запускает Word headless с `Document.Fields.Update()`), затем `inject-toc.py` переносит заполненный sdt-блок TOC + закладки `_TocNNN` в основной docx.
- **Источник:** `mf-8lo`, `mf-a3w`.

### B.10 Изображения смещены вправо / разного размера
- **Фикс:** в `postprocess-docx.py` для каждого `<w:drawing>` установить выравнивание родительского параграфа = center, ширину = ≤ 14 см.
- **Источник:** `mf-tsj`.

### B.11 Титульник как markdown ломает Pandoc
- **Симптом:** попытка собрать титул из markdown div-ов с custom-style давала кривую вёрстку.
- **Фикс:** титульник вынесен из markdown — генерируется отдельно через `inject-titlepage.py` (python-docx), вставляется в начало готового docx через постпроцесс.
- **Источник:** `mf-fo5`.

### B.12 Битые ссылки на assets
- **Фикс:** `build-docx.sh` перед сборкой проверяет каждый `![](path)` и абортит при битых ссылках. `--check` режим без сборки.
- **Источник:** `mf-2vs`.

## C. Сводная audit-таблица для скрипта

| Что проверить | Команда |
|---|---|
| em-dash | `grep -c '—' text/*.md` = 0 |
| `[0]` | `grep -c '\[0\]' text/*.md` = 0 |
| `**bold**` вне Реферата | `grep -lE '\*\*' text/0[1-8]-*.md` = empty |
| ГОСТ 7.32/7.1/7.0.5 | `grep -i 'ГОСТ 7\.' text/*.md docs/sources.md` = 0 |
| прогноз-слова | `grep -ic 'прогноз\|ожидае\|литературн' text/*.md` ≈ 0 |
| alt-text у картинок | `grep -c '!\[[^]]' text/*.md` = 0 |
| длинные абзацы | python-скрипт: 0 абзацев >200 слов |
| таблицы без [N]/собств | python: каждая `^Таблица N – ...` имеет `[N]` или "собственн" |
| таблицы без упоминания | python: `^Таблица N` имеет `таблиц[еау] N` выше |

## Как скилл это применяет

1. **На `init`:** этот файл копируется в `docs/lessons-learned.md` нового проекта. CLAUDE.md.template ссылается на него: "перед написанием секции — прочитать docs/lessons-learned.md".
2. **На `audit`:** проверки из раздела C запускаются автоматически, отчёт по нарушениям.
3. **На `build`:** все правила раздела B уже встроены в pipeline (build-docx.sh + apply-stosgau-styles.lua + postprocess-docx.py). Агент в этом не участвует.
